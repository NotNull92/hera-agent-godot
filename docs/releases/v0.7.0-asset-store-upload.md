## Detailed Description

Paste verbatim into the Asset Store **Detailed Description** field (Markdown, 4408 / 20000 chars).

```markdown
**Hera Agent Godot** lets AI coding agents inspect and control a **live Godot 4.7+ editor** in real time — instead of guessing scene structure from stale training data, the agent acts on the *real* editor and checks the result.

This Asset Library entry is the **editor addon** (the in-editor bridge). The companion **`hera` CLI** lives on GitHub and is what the agent actually runs.

## Why a CLI, not MCP?

Godot already has a healthy MCP-addon ecosystem — Hera makes the opposite bet on purpose. MCP servers pay for breadth in tokens: dozens to 100+ tool schemas plus verbose JSON responses sit in the agent's context every turn.

Hera delivers **MCP-grade reach over the live editor as a compact-JSON-by-default CLI** — one command per action, minimal tokens, and it works with anything that can run a shell command (pipes, `batch`, CI, any agent), not just MCP clients. The agent loads **zero tool schemas**: the only resident surface is one small cacheable doc (~1k tokens) that stays flat no matter how many commands exist, versus an estimated ~4k–31k resident tokens for sampled Godot MCP servers.

## What the agent can do

- **Read the editor** — live status, scene tree, node property dumps, open scenes, editor selection, and ClassDB inspection.
- **Edit the scene** — add / set / remove nodes, attach & detach scripts, wire resources, connect & disconnect signals. Every write is a single undoable step (Ctrl+Z).
- **Scripts & project** — create scripts, make folders, list and scan project files, inspect and refresh resources and UIDs.
- **Run GDScript** — evaluate an expression against the edited scene through an `EditorInterface` proxy.
- **Drive a running game** — inspect and mutate the runtime tree, call helper methods, click controls semantically, read the runtime UI tree, and assert on state.
- **Proof-first QA** — inject real input, confirm what was actually delivered, capture and analyze screenshots, and run requirement-covered QA scenarios from a file.
- **Diagnostics** — tail the Godot log with error/warning filters and summarize problems.

## New in v0.7.0 — Game Feel & runtime input

- **Game Feel Mode** — a dedicated Hera editor panel with live bridge status and persistent settings agents read before starting gameplay or UI work.
- **Bundled `game_feel` knowledge base** — query concrete topics (control feel, screen shake, hit stop, camera, sound, particles, reward feedback, UI bars, accessibility, ethical feedback) plus `guidance game-feel`, instead of vague "juice" prompts.
- **Runtime input injection** (`game input`) — mouse press/release/click/move, key press/release, InputMap action press/release with strength, text input, modifiers, and viewport routing for focused Control input.
- **Input diagnostics** (`game input-log`) — prove which runtime inputs were actually delivered: click coordinates, button state, short/long press classification, key names, modifiers, and active keys/buttons.
- **Requirement-covered QA** — `game qa --file` scenarios can include `game.input` and `game.input_log` steps, exercising real player keyboard, mouse, touch-style, and action paths rather than helper-only code.

## Command surface

`status`, `instances`, `run` / `stop`, `scene`, `editor`, `script`, `project`, `classdb`, `node` (read + write + resource/script wiring), `signal`, `resource`, `game` (runtime inspect + input + input-log + set/call/click + assert + QA + screenshot), `guidance`, `game_feel`, `output`, `diagnostics`, `eval`, `screenshot`, `batch`, and `smoke` — with `--json` (pretty) and `--ids` (paths only) output modes; compact JSON is the default.

## How it works

The addon is an `@tool` `EditorPlugin` that runs a localhost HTTP server and advertises the editor to the CLI via `~/.hera-agent-godot/instances/`. The Go CLI discovers the editor and sends one compact JSON request per command, executed on the editor main thread through `EditorInterface`. No cloud and no account — everything stays on localhost.

## Install

1. Use a **Godot 4.7+** build.
2. Copy `hera_agent_godot/` into your project's `res://addons/`.
3. Enable **Project → Project Settings → Plugins → Hera Agent Godot**.
4. Grab the `hera` CLI from GitHub (one-line installer or `go build`) and start driving the editor from your agent.
```

## Changelog

- Added a dedicated `HeraAgent` editor main-screen panel backed by shared Hera settings. The panel now exposes live bridge status, Hera branding, and the persistent Game Feel Mode controls agents use before starting gameplay or UI work.
- Added a bundled `game_feel` knowledge base and CLI/tool surface. Agents can query concrete topics such as control feel, screen shake, hit stop, camera, sound, particles, reward feedback, UI bars, accessibility, and ethical feedback instead of relying on vague "juice" prompts.
- Added gameplay-wide `guidance game-feel` output alongside existing UI guidance. The response now separates gameplay feel from Control layout work and returns compact QA patterns for runtime verification, accessibility options, Honest Juice, and observable feedback evidence.
- Added runtime input injection with `game input`: mouse press/release/click/move, keyboard press/release, InputMap action press/release with strength, text input, modifiers, physical key routing, and viewport routing for focused Control input.
- Added `game input-log` diagnostics so QA can prove which runtime inputs were actually delivered. The log tracks click coordinates, button state, short/long press classification, key names, modifiers, active keys, and active mouse buttons.
- Expanded `game qa --file` scenarios with `game.input` and `game.input_log` steps, letting requirement-covered QA drive primary keyboard, mouse, touch-style, and controller/action paths instead of validating only helper-only code paths.
- Strengthened the runtime inspector and viewport action bridge so live game QA can combine semantic clicks, injected input, deterministic helper calls, runtime UI reads, screenshots, and diagnostics in one ordered proof flow.
- Added Game Feel-aware editor hints: when Game Feel Mode is enabled, relevant `node add` operations can return compact `agent_hint` topics that point agents toward the matching `game_feel` guidance.
- Promoted v0.7.0 prompt-game QA findings into reusable guidance: ordered state-changing QA, direct affected-scene parsing when runtime registration is empty, current-project helper discovery, hidden-state precondition resets, deterministic helper quiescence, and stable inspection handoff states.
- Added visual/layout QA rules from hands-on prompt-game cycles: bounded child visuals for pieces/markers/tokens, derived grid insets, stable interactive frames with animated child content, live viewport-sized playfields, readable contrast, and parent/child rect verification through runtime UI tree reads.
- Added Game Feel verification rules from hands-on testing: require perceivable multi-channel feedback, expose target/channel/duration/intensity state, verify screen shake or board motion at the intended scope, and avoid treating feature-list strings or booleans as proof of player-visible feel.
- Added route/path QA rules for games with lanes, rails, patrols, projectiles, or tower-defense paths: drawing and movement should share one smoothed authoritative geometry, and screenshots plus live movement steps should prove entities stay aligned with the rendered path.
- Expanded the GDScript agent guide with typed-copy helpers for `Dictionary`/`Variant` arrays, style-resource separation from frame loops, selector/label synchronization after programmatic state changes, and direct runtime helper path discovery from the installed addon.
- Added `docs/reports/v0.7.0.md`, capturing the prompt-game QA cycles that produced these reusable rules, including UI consistency, keyboard input, Game Feel salience, screen shake scope, path smoothing, and runtime helper reliability findings.
- Refreshed the English, Korean, and addon README files so the public documentation matches the current v0.7.0 addon baseline and its new Game Feel and runtime input QA surfaces.
