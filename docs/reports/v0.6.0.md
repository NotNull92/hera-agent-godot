## breakout

### Stale Hera game processes blocked runtime QA

- Context: `go run . game screenshot --analyze` and `go run . game ui tree` initially refused to run because many stale game PIDs were still attached to `res://scenes/Main.tscn`.
- Impact: Runtime verification could not start until those old instances were cleared, so the first capture pass was delayed.
- Trace: `go run . game instances` returned 16 PIDs for the same scene. After stopping those processes, `go run . game instances` returned empty, and `go run . run --current --wait` succeeded.
- Improvement: Add a stronger preflight cleanup step before capture-heavy QA runs, or make the workflow stop stale runtime instances automatically.
- Status: Resolved for this cycle.

### First serve can end the ball quickly without paddle input

- Context: After `go run . game click --text Launch`, the UI switched to `Ball in play.` and the launch button disabled, but the next runtime screenshot showed `score 0010`, `lives 2`, and `Life lost. Press Launch to continue.`
- Impact: The game is playable and the launch path works, but the initial serve is still a bit unforgiving if the player does not move the paddle immediately.
- Trace: The runtime UI tree confirmed the launch-state transition, then the screenshot at `C:/Users/PC/AppData/Roaming/Godot/app_userdata/Hera Agent Godot (dev)/hera_game_screenshots/latest.png` showed the reduced life count and updated status message.
- Improvement: Widen the initial launch angle slightly or add a short post-launch grace window so the first rally is easier to catch.
- Status: Open.

### HUD panel container over-expanded until the outer panel stopped auto-sizing

- Context: The first Breakout HUD pass used `PanelContainer`, which caused the right-side panel to stretch to `height: 2843` in the live UI tree because the spacer inside the `VBoxContainer` kept expanding the container's minimum size.
- Impact: The sidebar content was technically present but no longer matched the viewport, so the live layout was misleading even though the game logic itself was running.
- Trace: `go run . game ui tree` first reported `RightPanel` at `299x2843`. After switching the outer node to `Panel` and reloading `res://scenes/Main.tscn`, the same command reported `RightPanel` at `299x600`, and `go run . game screenshot --analyze` returned `possible_clipping: false`.
- Improvement: Use a fixed panel for the outer HUD shell when the inner column needs an expanding spacer, and keep container-driven sizing inside the shell instead of on the shell itself.
- Status: Fixed for this cycle.

### Launch/restart flow works, but the first rally is still aggressive

- Context: After `go run . game click --text Launch`, the live HUD updated through a rally, then returned to `Ready. Launch the ball or press Enter.` with `Score: 330` and `Lives: 2`. Clicking `Restart` reset the HUD back to `Score: 0`, `Lives: 3`, and the ready message.
- Impact: The game is playable end to end and the reset flow works, but the first rally clears a lot of bricks and costs a life quickly, which makes the opening feel sharp rather than forgiving.
- Trace: `go run . game ui tree` showed the score/lives/status updates before and after `Restart`, `go run . game screenshot --analyze` stayed `nonblank: true` with `possible_clipping: false`, and `go run . output --type error --lines 120` remained clean.
- Improvement: If the opening pacing needs to be gentler, lower the initial serve speed or widen the paddle catch window a bit more.
- Status: Open.

## othello

### Control layout needed a fixed-size board wrapper to stay visible

- Context: The first Othello pass initially let `VBoxContainer`/`CenterContainer` expansion push the board and sidebar far below the viewport, even though the scene logic was loading.
- Impact: Replacing the centering container with a fixed-size board wrapper kept the 8x8 grid visible in the live game window and left the right-side controls reachable for QA.
- Trace: The live `game ui tree` settled on the board at `506x506`, `game screenshot --analyze` reported `nonblank: true` and `possible_clipping: false`, and the visible window showed the starting four pieces plus legal-move markers.
- Improvement: For board games in `Node2D`-hosted Control layouts, prefer fixed-size wrappers over nested expansion containers unless the board truly needs to stretch.
- Status: Resolved for this cycle.

### Undo had to snapshot before the AI reply to keep turn state sane

- Context: The Othello flow includes a human black move, an automatic white AI reply, an Undo button, and an AI ON/OFF toggle. That meant the undo stack had to capture state before each half-move, not just before a full turn.
- Impact: After one black move and the auto white reply, Undo rolled the board back to the previous half-turn cleanly, and toggling AI OFF immediately switched the right panel to `AI OFF` without forcing another reply.
- Trace: In live QA, the board advanced from `Black 2 / White 2` to `Black 3 / White 3`, Undo returned it to a `White to move` state with `Black 4 / White 1`, and clicking the AI toggle changed the button label to `AI OFF`.
- Improvement: Keep snapshot boundaries aligned with the interaction model the user sees, especially when an automated opponent can act between human inputs.
- Status: Resolved for this cycle.
