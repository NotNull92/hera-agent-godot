## hera-token-usage

### Runtime UI tree output needs scoped reads for larger scenes

- Context: Hera keeps fixed context cost low by using a shell CLI and compact JSON instead of exposing many MCP tool schemas every turn. In the current live project, compact output measurements were `status` 214 chars (~54 tok), `guidance ui` 646 chars (~162 tok), `scene tree` 116 chars (~29 tok), `--ids scene tree` 2 chars (~1 tok), `node get .` 792 chars (~198 tok), `game screenshot --analyze` 510 chars (~128 tok), and `game ui tree` 7852 chars (~1963 tok).
- Impact: The default surface is token-efficient for normal editor/runtime checks, but full runtime UI tree dumps can become expensive as Control-heavy scenes grow. Repeated `game ui tree` calls are the main token-risk path observed so far.
- Trace: The previous UI-heavy runtime test scene had only 36 visible Control entries, yet `game ui tree` was already about 7.8 KB. After implementation, full `game ui tree` measured 7858 bytes, while `game ui tree --type Button --fields name,path,text,disabled` measured 2803 bytes. `game ui tree --text Restart --fields name,path,text,rect,disabled` returned a single control, and `node get . --prop script` / `node get . --props visible,position` returned compact selected editor properties.
- Proposed change: Added scoped runtime UI reads such as `hera game ui tree --path <node>`, `--depth N`, `--fields name,path,text,rect,disabled`, `--type Button`, and `--text Restart`. Also added editor-side `node get --prop/--props` parity with runtime `game node get --prop/--props`.
- Patch note candidate: Reduced token usage for large UI QA by adding filtered Hera reads for runtime Control trees and focused editor node property inspection.
- Status: Implemented.

## hera-precision

### Live editor/runtime precision is strong, but deterministic QA needs first-class ergonomics

- Context: Hera reads the actual open Godot editor and runtime process instead of inferred scene structure. Editor commands resolve nodes through the edited scene root, mutation paths use `EditorUndoRedoManager`, runtime requests are PID-scoped through heartbeat/request files, and `game click --text/--node` resolves visible `Control` targets before injecting viewport input.
- Impact: Precision is high for concrete state checks: `node get . --prop script` returned `<null>` after rollback, invalid editor property reads failed with `node has no property: does_not_exist`, `game instances` returned empty after cleanup, and runtime UI QA in earlier prompt cycles caught real layout/state issues such as oversized panels, disabled buttons, status updates, and clipping.
- Trace: Code paths reject ambiguous or invalid targets rather than guessing: editor `node set` checks property metadata and coerces to the target type; runtime `game node get` can read nested properties, dictionary/array members, metadata, and vector/rect members; `game click` rejects hidden, disabled, empty-rect, off-viewport, or non-Control targets; screenshot analysis reports `nonblank`, edge content, and `possible_clipping`. After implementation, `game qa discover` returned an empty method list for the clean root, rejected `/root/Main/DoesNotExist`, and found a temporary probe's `qa_state()` plus `qa_step(seconds = 0.0)` before the probe was detached and deleted.
- Limitation: For timer-driven or hidden-state games, precision still depends on each generated game exposing ad hoc QA helpers such as pause, state dump, forced step, or test-target lookup. Reports repeatedly show this pattern across Snake, Top Down Dodger, Tower Defense, and hidden-information games.
- Proposed change: Add a documented QA-helper contract or scaffold for generated Godot games: `qa_state()`, `qa_pause(bool)`, `qa_step(seconds/frames)`, and optional domain helpers. Implemented `game qa discover [path]` to list callable `qa_*` methods on the runtime root or a target node, so agents can find deterministic hooks without reading the script first.
- Patch note candidate: Improved runtime QA precision by adding `game qa discover [path]`, which exposes callable `qa_*` helpers before agents call state-changing runtime methods.
- Status: Implemented.

## implementation-accuracy

### Prompt requirements need machine-checkable QA coverage, not only prose confirmation

- Context: Previous game implementation cycles verified live behavior with `game ui tree`, semantic clicks, screenshots, diagnostics, and ad hoc `qa_*` helpers, but the final evidence was still recorded mostly as prose. That makes it easy to prove individual interactions while still missing one explicit user requirement from the original prompt.
- Impact: `game qa --file` now supports a stricter object format with top-level `requirements` and per-step `covers`. A scenario fails if a declared requirement is not covered by any step, and the final result only counts requirements covered by successful steps.
- Trace: A real scenario with `requirements:["diagnostics-clean"]` and a `diagnostics` step covering it returned `ok:true`, `requirements_covered:["diagnostics-clean"]`, and `requirements_missing:[]`. A second scenario that also required `launch-flow` but had no covering step failed with `requirement "launch-flow" is not covered by any step`.
- Limitation: This does not magically extract requirements from free-form prompts; the agent still has to translate the user's prompt into concrete requirement IDs. It does prevent the QA artifact from claiming completion when those IDs are not mapped to executable checks.
- Proposed change: Use requirement-covered QA scenarios for prompt implementation tests, especially when a prompt includes multiple mechanics, UI states, reset flows, or accessibility/game-feel requirements.
- Patch note candidate: Added requirement coverage metadata to `game qa --file`, allowing QA scenarios to fail on uncovered user requirements and report covered/missing requirement IDs.
- Status: Implemented.

## efficient-methodology

### Prompt-driven game work should use a requirement-led QA loop

- Context: The repeated game prompt cycles showed the same waste pattern: the agent implements first, then reconstructs requirements and evidence afterward. That creates extra runtime passes, large UI dumps, ad hoc QA helper discovery, and prose-only claims that are hard to audit.
- Impact: Added `docs/GAME_PROMPT_WORKFLOW.md` as the recommended workflow: extract requirement IDs first, design deterministic `qa_*` hooks while building, use narrow runtime reads, prove requirements with `game qa --file` `requirements`/`covers`, then report only observed evidence.
- Trace: The workflow directly composes the earlier improvements: scoped `game ui tree` for token control, `game qa discover` for deterministic helper discovery, `game qa` requirement coverage for prompt adherence, `screenshot.runtime`/`diagnostics` for final observable checks, and `docs/reports/v0.6.0.md` for patch-note evidence.
- Limitation: This is a process-level efficiency gain rather than an automatic planner. The agent still has to translate each prompt into requirement IDs, but once it does, Hera can fail missing coverage mechanically.
- Proposed change: Use the workflow for every future `docs/tests` game prompt cycle and treat a passed requirement-covered QA scenario plus clean diagnostics as the minimum completion bar.
- Patch note candidate: Added a prompt-driven game workflow that turns free-form prompts into requirement-covered Hera QA scenarios, reducing repeated UI dumps and post-hoc verification work.
- Status: Implemented.
