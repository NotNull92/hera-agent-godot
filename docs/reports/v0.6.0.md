- Context: The `Main.tscn` save error came from overlapping external file edits and the live Godot editor's own save flow on the same scene.

## memory-match

### Headless Game Process Scene Lock

- Context: The repeated Godot "Cannot write file" modal was caused by an extra windowless `Godot_v4.7-stable_win64` game process still running for `res://scenes/Main.tscn`.
- Impact: Godot reported the scene as locked or unwritable even though `scenes/Main.tscn` had normal file attributes and the user had write permission.
- Trace: `Get-Process` showed the editor process plus a second Godot process with no window handle; `Get-CimInstance Win32_Process` showed that process was launched with `--scene res://scenes/Main.tscn`, and an exclusive ReadWrite open test reported the file as locked until that process was stopped.
- Improvement: Before editing or saving a scene through Hera, stop the running game with `hera-agent-godot.exe stop --wait`; if the modal appears again, check for windowless Godot processes and terminate only the stale play process, then verify the scene can be opened with exclusive ReadWrite access before continuing.
- Status: adopted

### Single-Writer Scene Workflow

- Context: `scene open`, `scene save`, direct `.tscn` edits, and running game processes can all touch the same scene file if invoked too closely together.
- Impact: Parallel or overlapping operations create false permission errors and make it unclear whether the live editor, disk file, or runtime process owns the current scene state.
- Trace: The lock disappeared after pending Hera commands completed, the stale game process was stopped, and `scenes/Main.tscn` passed an exclusive file-open check.
- Improvement: Treat scene work as a single-writer workflow: stop runtime first, apply file edits, reopen the scene, attach or detach scripts through Hera if needed, save once, then run the game. Do not run `scene open` and `scene save` in parallel.
- Status: adopted

## othello

### Runtime Undo Semantics With AI Turns

- Context: The Othello prompt required an Undo button while White can be controlled by AI.
- Impact: Undoing only the AI move left the game on White's turn with AI enabled, which made the human-facing Undo feel stuck because the board was no longer in a ready Black-turn state.
- Trace: Hera runtime QA played Black at `(2, 3)`, waited for White AI, then called `_qa_undo`. The first implementation returned `current_player=White`; the fix makes AI-enabled Undo roll back both the AI move and the preceding human move when the board is back on Black's turn. The verified result returned `black_score=2`, `white_score=2`, `current_player=Black`, `legal_moves=4`, and `history=0`.
- Improvement: For turn-based games with AI, define Undo granularity from the human player's perspective and verify it after the AI response, not immediately after the human move.

### Web Rule Cross-Check Caught AI Pass Edge Case

- Context: After the first Othello implementation, I checked external Othello/Reversi rule references before rollback. The relevant rule is that a player with no legal move must pass, and the opponent continues; the game ends only when neither player can move. References used: World Othello Federation official rules (`https://www.worldothello.org/about/about-othello/othello-rules/official-rules/english`) and US Othello Association tournament rules (`https://usothello.org/usoa-tournament-rules/`).
- Impact: The broad prompt included automatic passes, but the implementation only updated the UI in the pass branch. If Black had no move after a White AI move while White still had a legal move, the AI would not automatically continue.
- Trace: Added a runtime pass QA scenario in `scripts/othello.gd`; before the fix this path would leave White waiting after Black passed. After the fix, `_qa_force_pass_to_ai` produced `status=Black has no legal moves and passes.`, then White AI played `(0, 2)` and the runtime state became `game_over=true`, `white_score=3`, `black_score=0`, `status_text=Game over. - Improvement: For classic board games, include a web-backed edge-case checklist before rollback: mandatory pass/skip-turn behavior, repeated AI turns after a pass, terminal state only after both sides are stuck, and end-state scoring. Do not treat "pass handled" as verified unless both human-pass and AI-pass branches have been exercised.
