### Primary input schemes need live input QA, not helper-only validation

## tic-tac-toe-ai

### Semantic clicks on toggle labels must account for mutable UI state

- Context: During runtime QA, a state-mutating helper left a toggle button in its alternate state. A later semantic click targeted the old text label and failed even though the control was present and functional under its current label.
- Impact: `game click --text` is useful for user-facing controls, but it is brittle for toggles, mode switches, and stateful buttons whose text changes after helper calls or previous interactions. This can create false negatives in QA and obscure whether the game or the test sequence is wrong.
- Trace: After a draw-state helper left the AI toggle showing `AI OFF`, `go run . game click --text "AI ON"` failed with `control with text not found: AI ON`. Clicking the current label, `go run . game click --text "AI OFF"`, hit `/root/Main/TicTacToeUI/Sidebar/SidebarContent/AIButton` and `qa_state()` returned `ai_enabled:true`.
- Improvement: For stateful controls, either reset to a known baseline before semantic text clicks, read the runtime UI tree first and click the current label, or prefer a stable node path when verifying the control behavior itself.
- Status: Added as a reusable runtime QA rule.

### Sibling gameplay panels need shared geometry and density budgets

- Context: A user hands-on visual review caught a board-plus-sidebar UI that technically fit on screen but did not read as a coherent surface. The board panel and sidebar started at the same Y position, but their outer heights differed by more than 100 px and the sidebar content density forced the panel to grow independently.
- Impact: Automated screenshot checks can report `possible_clipping:false` while the UI still looks mismatched. Gameplay panels that function as one composed tool need matching outer geometry, balanced empty space, and a content density budget that prevents one panel from visually overpowering the others.
- Trace: Runtime UI rects showed `BoardPanel` as `406x406` and `Sidebar` as `310x515`. ## tower-defense-lite

### Runtime UIs must not force a stale viewport size

- Context: Visual QA found that a generated runtime UI drew to a hardcoded `1024x600` surface while Hera captured a `1152x648` viewport. The unused screen area appeared as blank grey margins, and the path end/base overlapped the sidebar boundary.
- Impact: A game can be mechanically playable while looking broken if its draw surface, HUD geometry, and runtime capture viewport disagree. Hardcoded viewport resizing inside gameplay scripts also fights the editor/runtime environment and makes screenshot evidence unreliable across machines or viewport sizes.
- Trace: The first screenshot showed high edge content (`edge_content_ratio:0.4847`) and visible grey margins on the right and bottom. The fix removed the runtime viewport-size assignment, drew the background using `get_viewport_rect()`, moved the field and sidebar into explicit padded rects, widened the sidebar content area, and moved the path endpoint/base away from the HUD. Fresh screenshot analysis returned `possible_clipping:false` and reduced edge content to `0.0208`.
- Improvement: For generated game UIs, derive the drawn background from the live viewport and keep map/playfield/HUD rects inside explicit padded layout bounds. Avoid setting `get_viewport().size` from a gameplay script unless the prompt explicitly requires fixed-resolution output.
- Status: Added as a reusable viewport/layout QA rule.
